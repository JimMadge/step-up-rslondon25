Why sensitive data?

- Research uses data - Research can produce results that benefit society - Some data useful for research is sensitive including - Medical data - Administrative data - Financial data - If we don't use it, we loose the benefits - **LINK:** But using sensitive data can be difficult

Why trusted research environments?

- Justified privacy and security concerns - TREs are a solution to that conflict - Secure environments with controls to prevent unauthorised data egress - Make more sensitive data available for research - Compared to sending data to researchers - Prevents unauthorised disclosure - Trust is more distributed (enables 5 safes) - **LINK:** Some work requires lots of computational power

Why HPC?

- Parallel processing - Very fast interconnects - Large, parallel filesystems - Large, fast storage - AI in particular - Large numbers of GPUs - RDMA - Fast GPU-GPU interconnect - Expense is justified for a shared system (maintain high throughput)

TREs are not HPC

HPC is not a TRE

FRIDGE

Shared responsibility

- **Challenge**: How to make HPC provider comfortable with high-risk data - **Key point**: Extend TRE governance to an enclave on the HPC system - Keep the responsibility and governance with the TRE operator - Protect the HPC provider from data protection questions - Avoids the need for complex data sharing agreements - **Link**: This informs the design! - HPC providers responsbility is - provide a secure space to build a TRE - **not** handle, process or transfer sensitive data

Job dispatch

- More abstract interface - Users submit workflow definition

FRIDGE architecture

- TRE tenant, secured/configured by HPC provider (link back) - Connection to external trusted resource (S2S VPN) - Sensitive data on HPC, close to compute - Job submission interface - "Buckets" for intermediates, results - Advantages - Considered part of existing TRE, under it's governance - Outputs do not need to be classified, they go to a TRE - Minimal services running on HPC - Promote reproducibility with workflows - Control of allowed container images - Disadvantages - "I want a notebook" - "I want a desktop" - Probably slow for iterative, exploratory work - Learning curve for packaging code in containers, workflow format

FRIDGE implementation

- Jobs - Argo Workflows - Containers - Harbor - Data - Minio (S3) - PVCs (encryption from K8s secrets) - Network policy - Cilium - Reverse proxy and TLS - Nginx - Cert manager - VPN - Wireguard - Secrets - etcd (?)

Lessons

- Governance is difficult with multiple parties, don't solve it unless you have to - K8s is becoming a standard base for private-cloud-HPC, and TRE builders (K8TRE, UCL, Lancaster)
