Why sensitive data?

- Research uses data - Research can produce results that benefit society - Some data useful for research is sensitive including - Medical data - Administrative data - Financial data - If we don't use it, we loose the benefits - **LINK:** But using sensitive data can be difficult

Why trusted research environments?

- Justified privacy and security concerns - TREs are a solution to that conflict - Secure environments with controls to prevent unauthorised data egress - Make more sensitive data available for research - Compared to sending data to researchers - Prevents unauthorised disclosure - Trust is more distributed (enables 5 safes) - **LINK:** Some work requires lots of computational power

Why HPC?

TREs are not HPC

  • Top priority is security
  • Don't typically use HPC hardware
  • Scaling on cloud is expensive
  • Often designed for a single domain
  • Restrictive
- Focus on security controls above all else - Often build on cloud systems - Great ability for "X" as code (policy, infra, networking) - Performance hit through virtualisation - Cloud costs for HPC/GPU nodes are high, and availability may be low - Can be inflexible, designed for a small group of users or domain with only the tools they need

HPC is not a TRE

  • Designed for power/throughput, not confidentiality
  • Shared systems
    • Many users and projects
    • Jobs share resources/nodes
  • Public
  • No/little controls on data ingress and egress
- Shared systems, inherently. Need to make the most out of investment by keeping throughput high - Many users, not all of whom will have training or responsibility for sensitive data - Jobs may share nodes, possibility for attacks (without TEE) - Accessed over the internet - Designed to be easy to get data in and out (code and results!)

FRIDGE

A ready-to-use SATRE compliant TRE that can be deployed to AIRR

- Quickly settled on K8s - Self-managed K8s is a first-class offering on Dawn and Isambard-AI - Private cloud model

Shared responsibility

Need agreement between TRE operator and infrastructure provider

How do we assure the host, that they are not accountable?

Extend the governance of a TRE to cover a secure enclave on AIRR (or another system)

Strict isolation of a TRE tenancy on the host gives the confidence to to delegate information governance to the operator

TRE operator Infrastructure provider
Data processing Securing TRE tenancy
User access and management
Data ingress and merging Providing K8s with GPU passthrough and S2S VPN
Release of results
- **Challenge**: How to make HPC provider comfortable with high-risk data - **Key point**: Extend TRE governance to an enclave on the HPC system - Keep the responsibility and governance with the TRE operator - Protect the HPC provider from data protection questions - Avoids the need for complex data sharing agreements - **Link**: This informs the design! - HPC providers responsbility is - provide a secure space to build a TRE - **not** handle, process or transfer sensitive data

FRIDGE architecture

- TRE tenant, secured/configured by HPC provider (link back) - Connection to external trusted resource (S2S VPN) - Sensitive data on HPC, close to compute - Job submission interface - "Buckets" for intermediates, results - Advantages - Considered part of existing TRE, under it's governance - Outputs do not need to be classified, they go to a TRE - Minimal services running on HPC - Promote reproducibility with workflows - Control of allowed container images - Disadvantages - "I want a notebook" - "I want a desktop" - Probably slow for iterative, exploratory work - Learning curve for packaging code in containers, workflow format

FRIDGE implementation

- Jobs - Argo Workflows - Containers - Harbor - Data - Minio (S3) - PVCs (encryption from K8s secrets) - Network policy - Cilium - Reverse proxy and TLS - Nginx - Cert manager - VPN - Wireguard - Secrets - etcd (?)

Security

Inside K8s

  • RBAC
  • PSS
  • Network policy

Outside K8s

  • Site-to-site VPN
  • TRE tenancy isolation
    • Virtual machines and network
    • Encryption through out-of-band secrets
- Three prongs inside K8s - RBAC, pods which users run have a service account which prevents them from modifying K8s/interacting with the kubeAPI - PSS, pods have no privileges on the host (_e.g._ root, networking) except to use GPUs - Network Policy, traffic within the cluster is heavily restricted, only what is necessary - Securing the boundary - Running with TRE Tenant, VM, if VM is compromised the physical host is not and vnet restrictions still apply - Data is encrypted at rest, with keys only visible to the FRIDGE manager - Which means, - Users have very little privilege - They cannot access the host or modify the K8s spec - If they do somehow manage to compromise the VM, physical host is still protected, tenants are still isolated - If the host system is compromised, the data is encrypted with keys not available to root(?)

Lessons

  • Multi-party governance is difficult
  • K8s is becoming standard
  • Self-service HPC has arrived
  • Not all K8s implementations are the same
  • Don't reinvent the wheel
- Governance is difficult with multiple parties, don't solve it unless you have to - K8s is becoming a standard base for private-cloud-HPC, and TRE builders (K8TRE, UCL, Lancaster) - K8s implementations vary - Don't reinvent the wheel - We have built on a lot of mature FOSS projects - We would have little hope of making something functional and sustainable if we tried to build it from scracth
